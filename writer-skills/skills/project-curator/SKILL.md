# Project Curator — куратор файловой базы проекта

Этот навык — библиотекарь лор-базы. Он не сочиняет канон и не оценивает сюжет; он
следит, чтобы знание лежало там, где его найдут, и ровно в одном месте.

## Зачем это нужно (главный принцип)

База проекта живёт по правилу **«один факт — один файл, остальные ссылаются»**.
Со временем оно ломается само собой: раздел «Связи» в карточке персонажа
распухает до отдельной мини-главы, описание одной способности перерастает строку
в журнале, организация обрастает деталями внутри общего файла. Когда блок
перерастает свой дом, его надо **вынести в отдельный файл и оставить на его месте
короткий указатель со ссылкой** — тогда исходный файл снова читается, а
подробность живёт в одном месте, куда все ссылаются.

Это та же логика, по которой устроены сами навыки: наверху — короткая сводка,
вглубь — по ссылке. Цель куратора — держать базу в этой форме, не теряя ни факта
и ни одной связи.

## Золотое правило: показать → подтвердить → применить

**Куратор никогда не меняет файлы до подтверждения автора.** Сначала он собирает
весь план правок и показывает его в удобном для ревью виде, где автор отмечает, что
применять. Только отмеченное применяется. Это не формальность: автор — единственный
владелец канона, и любой перенос/слияние он должен увидеть до того, как оно
произойдёт. Подробности — в шаге 3.

## Рабочий процесс

Пять шагов, и шаг «Предложение» обязателен всегда — даже если просьба точечная
(«вынеси связи Наташи»): всё равно покажи мини-план и дождись «ок».

### 1. Карта проекта (scan)

Построй дерево папок и файлов с размерами и определи роль каждого файла (карточка
персонажа, файл мира, журнал/досье способности, сюжетный разбор, индекс,
справочник). Целевую таксономию и шаблоны смотри в `references/project-structure.md` —
читай его в начале работы.

```bash
cd "<корень проекта>" && for f in $(find . -name "*.md"); do \
  echo "$(wc -l < "$f") строк | $f"; done | sort -rn
```

### 2. Диагностика (diagnose)

Пройди по базе и собери кандидатов на правку. Пороги — **умеренные**: трогаем то,
что реально разрослось или лежит не там, мелочь не дёргаем.

**(а) Разросшийся блок → вынести.** Выноси раздел, если он перерос дом по любому
признаку: занимает больше ~40% файла **или** больше ~25 строк и самодостаточен; либо
описывает отдельную **сущность** со своим местом в таксономии (пара персонажей →
`Персонажи/Отношения/`, способность ГГ → `Способности_ГГ/Досье_*`, организация или
локация → подпапку в `Мир/`); либо один и тот же подробный блок начал дублироваться.
Короткую сводку (2–4 строки) и факт-указатель НЕ выносим — это и есть то, что
остаётся в исходнике.

**(б) Файл не в своей папке → переместить.** Сюжетные разборы, мастер-планы и правки
из корня → в `Сюжет/`; досье способности из корня → в `Способности_ГГ/` и т.п.
Перемещаем файл и чиним ссылки на него.

**(в) Битые ссылки — но сначала отличи memory.** Найди `[[вики-ссылки]]` и
относительные md-ссылки, ведущие в несуществующий файл/якорь. **Прежде чем звать
ссылку битой, проверь, не указывает ли она на файл памяти (memory)** — см. раздел
«Работа с memory» ниже. Memory-ссылки НЕ битые и НЕ чинятся. Настоящие битые
(цель — внутрипроектный файл, которого нет) либо почини (если цель
переименована/перемещена), либо вынеси в план с пометкой `(?)`.

**(г) Дубли и конфликты факта.** Один и тот же факт, прописанный содержательно в
нескольких файлах, — нарушение «один факт → один файл». Если версии совпадают —
предложи назначить источник и свести остальные к ссылке. Если версии **расходятся**
(конфликт) — не выбирай за автора: подними **интерактивный артефакт-резолвер** (см.
шаг 3) с диффом вариантов и выбором.

**(д) Отклонение карточки от шаблона.** Сверь файлы с шаблоном из
`references/project-structure.md` (статус-блок, маркеры 🟡🟢🔴, пометки `(?)`/`(!)`).
Отметь заметные расхождения, но не навязывай косметику.

**(е) Рассинхрон с memory.** Если факт в файле памяти и в файле проекта **разошлись**,
или memory выглядит избыточным (полностью покрыт файлом проекта) — вынеси это в план
как **выбор для автора**, не правя ничего молча. Правила — раздел «Работа с memory».

**(ж) Замеченные канон-противоречия → только флажок.** Если по ходу наткнулся на
смысловое противоречие в самом каноне (разные числа, несостыковка правил, силы) —
**не правь его**: канон не наша территория. Отметь находку в плане одной строкой и
посоветуй прогнать `plot-check` (логика/сюжет) или `feat-judge` (баланс сил). Куратор
сводит и связывает, но смысл фактов решает автор.

### 3. Предложение (propose) — показать на ревью, ждать «ок»

Собери все находки шагов 2(а–е) в один план и покажи его автору **в удобном для
ревью виде до любых правок**. Используй интерактивный артефакт-чеклист из
`assets/change_plan_template.html`:

1. Прочитай шаблон, подставь вместо плейсхолдера `/*PLAN_DATA*/` объект с пунктами
   (`id`, `type`, `summary`, `from`, `to`, `detail`).
2. Покажи его как виджет (в Cowork — через инструмент показа виджета; вызови сначала
   его инструкцию-`read_me` с модулем `elicitation`, затем покажи виджет). Кнопки
   шлют выбор автора обратно в чат через `sendPrompt`.
3. Если виджет недоступен — деградируй до текстового плана: сгруппируй по видам и для
   каждого переноса дай карточку правки:

```
[Перенос блока]
Откуда: Персонажи/Наташа-Романова.md → раздел «Связи» (~28 строк, ~45% файла)
Куда:   Персонажи/Отношения/Морозов-×-Романова.md  (дополнить существующий)
Почему: блок перерос карточку, описывает связь как самостоятельную сущность
Останется: 3-строчная сводка + «Подробнее → [[Морозов-×-Романова]]»
Индекс:  строка в Персонажи/Отношения/00_Индекс.md
```

Не применяй ничего, пока автор не отметит пункты и не подтвердит.

**Конфликты фактов — отдельный артефакт.** Для каждого расходящегося факта подними
резолвер из `assets/conflict_resolver_template.html`: подставь вместо
`/*CONFLICT_DATA*/` объект `{ fact, note, variants:[{source, content}] }` и покажи
виджетом. Автор увидит дифф (уникальные строки подсвечены), выберет файл-источник
**или** опишет своё решение в поле «свой вариант» («что сохранить, что отбросить»).
Его выбор вернётся в чат через `sendPrompt` — по нему и своди.

### 4. Применение (apply) — только подтверждённое

Для каждой одобренной правки соблюдай порядок, чтобы не потерять ни факта, ни
ссылки.

**Вынос блока:**
1. Создай файл-приёмник в правильной папке по шаблону (заголовок, статус-блок,
   обратные ссылки `[[...]]`). Если файл уже есть — дополняй, а не плоди дубль.
2. Перенеси **полный** блок дословно (правки канона — не наша задача).
3. В исходнике замени блок на сводку (2–4 строки) + `Подробнее → [[<файл>]]` в стиле
   проекта.
4. Обнови `00_Индекс.md` папки-приёмника.
5. Почини обратные ссылки в других файлах.

**Перемещение файла:** перенеси файл, обнови все ссылки на него (в индексах и
карточках) на новый путь/имя.

**Слияние конфликта/дубля:** оставь выбранный автором источник, в остальных файлах
замени подробное описание на ссылку. Если автор дал «свой вариант» — следуй его
инструкции дословно.

Используй файловые инструменты (Read/Write/Edit), а для перемещений — `bash`.
**Важно про кириллицу:** правь .md инструментами Read/Edit (или Python-replace), а не
`sed`/heredoc по живому файлу — побайтовые операции портят кириллицу. Не удаляй
контент — он переезжает, а не исчезает.

### 5. Проверка (verify)

После правок пройдись ещё раз: ни одна ссылка не сломалась (перепроверь `[[...]]` и
пути; memory-ссылки по-прежнему на месте и не тронуты); каждый затронутый
`00_Индекс.md` соответствует содержимому папки; исходные файлы читаются связно (на
месте выноса — сводка, а не дыра); ни один факт не пропал. Заверши кратким отчётом:
что перенесено/перемещено/слито/починено и что автор отложил.

## Работа с memory (файлы памяти)

В проекте есть файлы авто-памяти (обычно в папке `memory/` с `MEMORY.md`-индексом;
имена — kebab-case слаги вроде `bolezn-lor`, `abilities-log`). На них ссылаются как
`[[слаг]]` или инлайном `memory \`слаг\``. Эти файлы нужны ассистенту для контекста
между сессиями.

Правила:
- **Не считай ссылку на memory битой** и не «чини» её на внутрипроектный файл. Если
  `[[слаг]]` совпадает с файлом памяти — это рабочая ссылка.
- **Не редактируй и не удаляй memory-файлы молча.** По умолчанию — не трогай вовсе.
- Если факт в memory и в файле проекта **разошёлся** — вынеси автору **выбор**:
  (а) обновить memory под файл, (б) обновить файл под memory, (в) оставить как есть.
- Если memory выглядит **избыточным** (полностью покрыт файлом проекта) — можно
  **предложить удалить** его, но только с явного согласия автора. Сам не удаляй.
- Любое действие с memory идёт тем же путём «показать → подтвердить → применить».

## Чего НЕ делать

- Не редактируй сам канон (формулировки фактов, баланс сил, сюжет) — это территория
  автора и навыков `character-card` / `feat-judge` / `plot-check`.
- Не выноси мелочь ради дробления.
- Не применяй ничего без одобрения плана.
- Не теряй пометки `(?)` / `(!)` и статусы при переносе — они едут вместе с блоком.
- Не трогай memory-файлы без явного согласия.
